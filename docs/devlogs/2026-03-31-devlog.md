# Devlog - 2026-03-31

## 今日主题
今天的重点不是重做架构，而是在现有可运行版本上，把「评审报告 -> 对话式评审助手」这条链路做顺、做稳、做得更可用。

主要集中在三条主线：

1. 把评审助手从“能聊”推进到“能围绕具体 issue 对话”。
2. 修复 MiniMax 接入和评审异常场景，保证单次运行更稳定。
3. 持续优化前端体验，降低页面拥挤感和理解成本。

## 完成内容

- 助手页从普通补充面板，调整为独立的对话页面，并使用 hash 路由保留两个页面态：
  - `#/report`
  - `#/assistant`
- 页面内原有 `Step 1-5` 文案、切换助手按钮、冗余步骤入口都已清理，页面标题改为普通业务标题。
- 维度评分详情改成纵向卡片布局，支持逐个点击展开，默认信息密度更低。
- 问题清单的编号逻辑修复，聚合后的问题 ID 不再出现重复回绕。
- 后端报告为 issue 增加了稳定身份字段：
  - `display_id`
  - `issue_key`
- 助手对话链路改成显式状态返回，不再把模型异常伪装成正常回答。当前会区分：
  - `model`
  - `report_level`
  - `unavailable`
  - `error`
- 助手问题定位能力增强，支持按以下顺序解析 issue：
  - `issue_key`
  - `display_id`
  - 标题匹配
  - prompt 关键词兜底
- 前端报告页和助手页现在能通过稳定 issue 身份对齐，不再只依赖 `HIGH-1` 这类展示编号。
- 修复了 MiniMax `2064` 临时繁忙错误：
  - 维度评审会先自动重试一次
  - 若仍失败，会返回明确的“服务暂时繁忙，请稍后重试”提示
- 助手页视觉层级继续优化：
  - 删除“操作提示”冗余模块
  - 放大对话主面板和输入区
  - 压缩摘要卡权重
  - 为“上下文”“建议动作”增加展开引导标识
- 助手页右上角按钮从“重新评审”改成“返回报告”，避免误触发新评审。
- 新增了“单次运行工具”范围内的 gap closure 开发计划，明确后续不做工作台式能力，而是优先补齐证据链、问题闭环、导出建议、文本复核等单次运行能力。

## 关键问题与处理

### 1. 对话内容过于僵化

之前用户输入不同提示词时，经常得到相同的兜底句，无法围绕具体问题持续追问。

今天处理结果：

- 聊天服务不再只依赖当前选中 issue。
- 后端增加稳定 issue 身份和多级解析策略。
- 模型不可用时会明确暴露状态，而不是继续输出看起来像“正常回答”的兜底文本。

### 2. 评分与问题聚合不稳定

今天处理结果：

- 修复综合评分上限问题，避免总分超过 `10`。
- 修复聚合后 issue 编号重复问题。
- issue 现在同时保留稳定内部标识和展示编号。

### 3. 模型调用异常影响体验

今天处理结果：

- 修复 `.env` / runtime 配置读取相关问题。
- 增强 MiniMax 临时繁忙错误处理。
- 后端错误分类和提示更清晰，便于判断是 key、模型、超时还是服务繁忙。
